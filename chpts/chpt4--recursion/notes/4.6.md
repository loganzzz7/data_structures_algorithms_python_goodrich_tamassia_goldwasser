# 4.6 Eliminating Tail Recursion
- The main benefit of a recursive approach to algorithm design is that it allows us to succinctly take advantage of a repetitive structure present in many problems.
    - By making our algorithm description exploit the repetitive struecture in a recursive way, we can avoid complex case analyses and nested loops -> creating more readable algorithm descriptions while still maintaining good efficiency
        - **HOWEVER** one downside of recursive algorithms is memory usage, since in recursion each level has an activation record!
    - In general, the `stack` data structure can be used to convert a recursive algorithm into a nonrecursive one--via managing the nesting of the recursive structure ourselves rather than relying on the interpreter -> this makes it so that instead of having the interpreter create recursive activation records, we can dynamically decide what information we want to store in each recursive trace ourselves on the stack data structure
        - This way we can try to store the minimal information possible and reduce memory cost
- **Some forms of recursion may be entirely eliminated without any use of axillary memory**
    - i.e. **tail recursion**: a recursion is a **tail recursion** if any recursive call that is made from one context is the very last operation in that context with the return value of the recursive call (if any) immediately returned by the enclosing recursion--so the last thing before returning is call itself
        - By necessity, a **tail recursion** must be a **linear recursion**--since there is no way to make a second recursive call if you must immediately return the result of the first
            - i.e. `binary_search`
        - An example of recursion that isn't a tail recursion:
            - **factorial**: `return n * factorial(n - 1)`
            - This is not a tail recursion because an additional multiplication is performed after the recursive call is completed!
        - Any tail recursion can be reimplemented nonrecursively by enclosing the body in a loop for repetition and replacing a recurisive call with new parameters by a reassignment of the existing parameters to those values
            - i.e. `binary_search` reimplemented
            ```Python
            def binary_search_iterative(data, target):
                """Return True if target is found in the given list else False"""
                # init vars low n high -> so instead of binary_search(data, target, low, high) -> just iterative with while
                low = 0
                high = len(data) - 1

                while low <= high:
                    mid = (low + high) // 2
                    if data[mid] == target:
                        return True
                    elif target > data[mid]:
                        low = mid + 1
                    elif target < data[mid]:
                        high = mid - 1
                return False
            ```
            - i.e. `reverse_iterative(S)`
            ```Python
            def reverse_iterative(S):
                """reverse elements in S"""
                start, stop = 0, len(S)
                while start < stop - 1:
                    # do the reverse
                    S[start], S[stop - 1] = S[stop - 1], S[start]

                    # move onto the next pair
                    start, stop = start + 1, stop - 1
            ```
            - The values `start` and `stop` during each iteration gets updated and the loop exits when the "base case" is reached--when the loop condition is no longer true == there is only one or less elts in the range
