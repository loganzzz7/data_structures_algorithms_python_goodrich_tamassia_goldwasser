# 4.3 Recursion Run Amok
- Bad cases for recursion
    - i.e. **elt uniqueness problem**
        - Base case: n = 1, the elts are trivially unique
        - $n \geq 2$: the elts are unique iff the first $n - 1$ elts are unique, the last $n - 1$ elts are unique and the first and last elts are unique
        ```Python
        # recursive algorithm for solving elt uniqueness problem
        def unique3(S, start, stop):
            # there is only 1 item
            if stop - start <= 1:
                return True
            elif not unique3(S, start, stop - 1):
                # first part--excluding last elt--has dup
                return False
            elif not unique3(S, start + 1, stop):
                # second part--excluding first elt--has dup
                return False
            return True if S[start] != S[stop - 1] else False
        ```
        - if there is one elt -> `O(1)`
        - if more then it recurses with exponential calls of size `n - 1`
            - So worst case: $O(2^n)$
    - **An Inefficient Recursion for Computing Fibonacci Numbers**:
        - Example of bad recursion usage for fibonacci:
        ```Python
        def bad_fib(n):
            """return the nth fib number"""
            if n <= 1:
                return n
            else:
                return bad_fib(n - 1) + bad_fib(n - 1)
        ```
        - This is bad because:
            - Runtime of exponential time since each iteration requires double calls
    - **An Efficient Recursion for Fibonacci**:
        - Example of good recursion usage for fibonacci:
        ```Python
        def good_fib(n):
            """return pair of Fib numbers, F(n) and F(n - 1)"""
            if n <= 1:
                return (n, 0)
            else:
                (a, b) = good_fib(n - 1)
                return (a + b, a)
        ```
        - This is good because:
            - The runtime is $O(n)$ since in $n$ iterations there are only one call per iteration

## 4.3.1 Maximum Recursive Depth in Python
- **Infinite Recursion**:
    - If each recursive call makes another recursive call without a base case to end the bottom call -> results in infinite recursion
    ```Python
    def fib(n):
        # fib(n) equals fib(n)
        return fib(n)
    ```
    - This will result in a `RuntimeError` type error with message--`Maximum recursion depth exceeded`
        - A programmer should ensure that each recursive call progresses towards the base case--i.e. having the parameter value decrease with each call
            - However, to combat against infinite recursions, python made an intentional decision to limit the overall number of function activation that can be simultaneously active.
                - So even if the recursive algorithm is sound it may still terminate if it ends up running to 1000 iterations
            - But fortunately in python there is a way to dynamically reconfigured to change the limit -> use the `sys` module
        ```Python
        import sys
        old = sys.getrecursionlimit() # check the system set limit
        sys.setrecursionlimit(1000000) # set custom recursion limit!
        ```
