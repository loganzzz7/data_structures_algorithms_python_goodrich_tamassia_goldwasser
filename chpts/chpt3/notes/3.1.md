# 3.1 Experimental Studies
- If an algorithm has been implemented -> can stuy running time by executing it on various test inputs and recording the time spent during each execution
    - to do this -> use the Python `time` module

Example:
```python
from time import time

start_time = time()
# run algo
end_time = time()
elapsed = end_time - start_time
```

- The `time` function measures relative to what is known as the "wall clock"
    - Because many processes share use of a computer's **central processing unit (CPU)**, the elapsed time will depend on what other processes are running on teh computer when the test is performed.
        - Therefore, a fairer metric is the number of CPU cycles that are used by the algorithm
            - This can be determined using the `clock` function of the `time` module, but this still depends on the computer system so running the same algo with the same inputs multiple times may still arrive on different results
            - Python has a more advanced module `timeit` to help automate evals with repetition to account for variance among trials
- **Challenges of Experimental Analysis**:
    - It is hard to compare two algos unless running on the exact same hardware and software
    - Not possible to test on every single possible input -> leaves out the running times of inputs not included--tested
    - An algorithm must be fully implemented to run it and get the runtime

## 3.1.1 Moving Beyong Experimental Analysis
- **Counting Primitive Operations**:
    - Define a set of **primitive operations** such as:
        - Assigning an identifier to an objext
        - Determining the objext associated with an identifier
        - Performing an arithmetic operation
        - Comparing two numbers
        - Accessing a single element of a Python list by index
        - Calling a function (excluding operations executed within the function)
        - Returning from a function
    - Each operation translate to base assembly instructions so therefore counting primitive operations can roughly detail the runtime
- **Measuring Operations as a Function of Input Size**:
    - Associate each algorithm with a function f(n) that characterizes the number of primitive oeprations that are performed as a function of the inputside n
        - **Focusing on the worse case input** -> 
            - an algorithm may run faster on some inputs than it does on others of the same size, so we want to express the running time of an algorithm as the function of the input size obtained by taking the average overall possible inputs of the same size
            - However, an **average-case** analysis is typically quite challenging to do--requiring calculation of expected running time based on a given input dist -> therefore, running times are to be characterized by the **worst-case** running time
