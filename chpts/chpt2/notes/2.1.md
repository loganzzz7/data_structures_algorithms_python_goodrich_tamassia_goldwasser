# 2.1 Goals, Principles, and Patterns
- The main actors in oop are "objects". Ea obj is an instance of a class, each class presents to the outside world a concise and consistent view of the objects that are instances of this class, without going too much into detail or giving others outside too much access to the inner workings of the objects.
    - The class definition typically specifies `instance variables` aka `data members` and `methods` aka `member functions` that the object can execute
## 2.1.1 Object-Oriented Design Goals
- Software implementations should achieve `robustness, adaptability and reusability`
    - **Robustness**:
        - capable of handling unexpected inputs that are not explicitly defined for its application--catching unexpected input with `try except` 
    - **Adaptability**:
        - up and downward compatibility, `portability`--abilty of software to run with minimal changes on different hardware and operating system platforms
    - **Reusability**: 
        - The same code can be usable as a component of different systems in various applications.
## 2.1.2 Object-Oriented Design Principles
- Principles: **Modularity**, **Abstraction**, **Encapsulation**
    - **Modularity**:
        - Separating the code into modules!!
        - Modern software systems consist of several diff components that much interact correctly in order for the entire system to work properly -> different components should be well organized--into modules!!
        - Module: a collection of closely related functions and classes that are defined together in a single file of source code!! -> i.e. `math module`
    - **Abstraction**:
        - distill a complicated system down to its most fundamental parts -> describing parts of a system involves naming them and explaining their functionality!
        - **abstract data types** (ADTs): a model of a data structure that specifies the type of data stored, the operations supported on them and the types of parameters of the operations.
            - Specifies **what** each operation does **BUT NOT** **how** it does it!
            - Also known as the **public interface**
        - Python treats abstractions implicitly using **duck typing** -> as a interpreted and dynamically typed language, there is no "compile time" checking of data types in Python and no formal requirement for declarations of abstract base classes--unlike cpp, no need to specify `int x = 2` can just do `x = 2` and then python determins that x is int
            - Python supports abstract data types using a mechanism known as an **abstract base class** (ABC). An abstract base class cannot be instantiated--cannot directly create an instance of that class
            - defines one or more common menthods that all implementations of the abstraction must have
            - An ABC is realized by one or more concrete classes that inherit from the abstract base class
    - **Encapsulation**:
        - Different components of a software system should not reveal the interal details of their respective implementations!!
            - Encapsulation yields robustness and adaptability -> allows the implementation details of parts of a program to change without adversely affecting other parts -> easier to fix bugs or add new functionality with relatively local changes to a component
        - **PYTHON CONVENTION**: names of members of a class that start with a single underscore character--i.e. `_secret` are assumed to be nonpublic and should not be relied upon.
            - `_example` is protected--one underscore
            - `__example` is private--two underscores
## 2.1.3 Design Patterns
- A solution to a "typical" software design problem -> a general template for a solution in an abstract way that can be specialized for a specific problem
    - Consists of: 
        - name--identifies the pattern
        - context--describes how the pattern is applied
        - result--describes and analyzes what the pattern produces
    - **Algorithm Design Patterns**:
        - Recursion
        - Amortization
        - Divide-and-conquer
        - Prune-and-search
        - Brute Force
        - Dynamic Programming
        - Greedy Method
    - **Software Engineering Design Patterns**:
        - Iterator
        - Adapter
        - Position
        - Composition
        - Template
        - Locator
        - Factory Method
